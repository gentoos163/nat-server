# RustDesk 打洞流程详解（结合源码）

> 目的：对照 `rustdesk-server`（hbbs / hbbr）与 `rustdesk` 客户端的官方实现，逐报文拆解 NAT 打洞与
> Relay 回退流程，并在每一阶段标注本仓库 `nat-server/src/*` 中的对应位置，便于验证本地实现。
>
> 引用来源（已落在 `docs/upstream-refs/` 下，方便对照）：
> - 服务端：`rustdesk-server/src/rendezvous_server.rs`、`relay_server.rs`、`common.rs`
> - 客户端：`rustdesk/src/rendezvous_mediator.rs`、`src/client.rs`
> - 协议：`hbb_common/protos/rendezvous.proto`
>
> 阅读约定：
> - **A** = Controller / Caller，主动发起连接的一端（"我要远控对方"）
> - **B** = Controlled / Callee，被远控端，需要先在 hbbs 注册
> - **hbbs** = Rendezvous Server（默认 21115 TCP + 21116 UDP/TCP + 21117 NAT 测试 + 21118 WS）
> - **hbbr** = Relay Server（默认 21117 TCP + 21119 WS），打洞失败的兜底转发节点

## 0. 端口与协议总览

| 端口 | 协议 | 角色 | 用途 |
|------|------|------|------|
| 21115 | TCP | hbbs | NAT 测试 / `OnlineRequest`（心跳保活后端） |
| 21116 | UDP | hbbs | **核心**：`RegisterPeer / PunchHoleRequest / PunchHoleSent / LocalAddr / TestNatRequest` |
| 21116 | TCP | hbbs | 与 UDP 同协议的 TCP 替代通道（含 `RequestRelay / RelayResponse`） |
| 21117 | TCP | hbbr | 两路 TCP 配对中继（client 与 controller 都连过来，按 `uuid` 配对后双向转发） |
| 21118 | TCP | hbbs | WebSocket 版的 21116 |
| 21119 | TCP | hbbr | WebSocket 版的 21117 |

## 4. 阶段 (3)：hbbs 处理 `PunchHoleRequest` —— 核心调度

`rustdesk-server/src/rendezvous_server.rs:682` `handle_punch_hole_request`（**逐行注释**）：

```rust
async fn handle_punch_hole_request(
    &mut self,
    addr: SocketAddr,            // = addr_a，A 的公网映射
    ph: PunchHoleRequest,
    key: &str,
    ws: bool,
) -> ResultType<(RendezvousMessage, Option<SocketAddr>)> {
    let mut ph = ph;

    // ── 1) license key 鉴权 ─────────────────────────
    if !key.is_empty() && ph.licence_key != key {
        // 给 A 回 LICENSE_MISMATCH
        msg_out.set_punch_hole_response(PunchHoleResponse {
            failure: punch_hole_response::Failure::LICENSE_MISMATCH.into(), ..
        });
        return Ok((msg_out, None));
    }

    // ── 2) 查询 B 是否在 PeerMap ────────────────────
    if let Some(peer) = self.pm.get(&id).await {
        let (elapsed, peer_addr) = {
            let r = peer.read().await;
            (r.last_reg_time.elapsed().as_millis() as i32, r.socket_addr)
        };
        if elapsed >= REG_TIMEOUT /* 30s */ {
            // B 心跳超时，告诉 A 离线
            msg_out.set_punch_hole_response(PunchHoleResponse {
                failure: punch_hole_response::Failure::OFFLINE.into(), ..
            });
            return Ok((msg_out, None));
        }

        // ── 3) 局域网判定 + 强制 relay 判定 ─────────
        let peer_is_lan = self.is_lan(peer_addr);
        let is_lan      = self.is_lan(addr);
        let mut relay_server = self.get_relay_server(addr.ip(), peer_addr.ip());

        // ALWAYS_USE_RELAY 或 一端 lan/一端公网 → 伪造成 SYMMETRIC，强制 A 走 relay
        if ALWAYS_USE_RELAY.load(Ordering::SeqCst) || (peer_is_lan ^ is_lan) {
            if peer_is_lan { relay_server = self.inner.local_ip.clone() }
            ph.nat_type = NatType::SYMMETRIC.into();   // ← A 看到的 nat_type 被改写
        }

        // ── 4) 同内网检查（公网 IP 相同）────────────
        // 如果 A、B 公网 IP 相同，路由器一般不会回环转发，必须用 LAN 地址；
        // 此时不发 PunchHole，而是发 FetchLocalAddr 让 B 把内网地址告诉 A
        let same_intranet = !ws && (peer_is_lan && is_lan || {
            match (peer_addr, addr) {
                (SocketAddr::V4(a), SocketAddr::V4(b)) => a.ip() == b.ip(),
                (SocketAddr::V6(a), SocketAddr::V6(b)) => a.ip() == b.ip(),
                _ => false,
            }
        });

        let socket_addr = AddrMangle::encode(addr).into();   // ← A 的公网 addr 编码后给 B
        let mut msg_out = RendezvousMessage::new();
        if same_intranet {
            msg_out.set_fetch_local_addr(FetchLocalAddr {
                socket_addr, relay_server, ..Default::default()
            });
        } else {
            msg_out.set_punch_hole(PunchHole {
                socket_addr,                 // ← addr_a 经过 AddrMangle 编码
                nat_type: ph.nat_type,
                relay_server,
                ..Default::default()
            });
        }
        Ok((msg_out, Some(peer_addr)))   // ← 第二个返回值=要发给 B 的目标地址
    } else {
        // PeerMap 里没有这个 ID
        msg_out.set_punch_hole_response(PunchHoleResponse {
            failure: punch_hole_response::Failure::ID_NOT_EXIST.into(), ..
        });
        Ok((msg_out, None))
    }
}
```

## 5. 阶段 (4)：B 端收到 `PunchHole` —— 真正的"打洞动作"

`rustdesk/src/rendezvous_mediator.rs:616` `handle_punch_hole`：

```rust
async fn handle_punch_hole(&self, ph: PunchHole, server: ServerPtr) -> ResultType<()> {
    let mut peer_addr = AddrMangle::decode(&ph.socket_addr);   // ← 解出 addr_a
    let relay_server = self.get_relay_server(ph.relay_server);
    let relay = use_ws() || Config::is_proxy() || ph.force_relay;

    // ── 1) 服务端告诉我对方是 SYMMETRIC，或我自己是 SYMMETRIC，或被强制 relay
    //       → 直接走 relay，不浪费 RTT 去试 UDP 打洞
    if ph.nat_type.enum_value() == Ok(NatType::SYMMETRIC)
        || Config::get_nat_type() == NatType::SYMMETRIC as i32
        || relay
        || (config::is_disable_tcp_listen() && ph.udp_port <= 0)
    {
        let uuid = Uuid::new_v4().to_string();
        return self.create_relay(ph.socket_addr.into(), relay_server, uuid, server,
                                 true, true, socket_addr_v6, control_permissions).await;
    }

    // ── 2) 构造 PunchHoleSent，等下要发给 hbbs，并让 hbbs 转发给 A
    let msg_punch = PunchHoleSent {
        socket_addr: ph.socket_addr,             // ← 原样回带 addr_a，方便 hbbs 路由
        id: Config::get_id(),
        relay_server,
        nat_type: nat_type.into(),               // ← B 自己的 NAT 类型，传给 A
        version: crate::VERSION.to_owned(),
        socket_addr_v6,
        ..Default::default()
    };

    // ── 3a) UDP 打洞分支 ──
    if ph.udp_port > 0 {
        peer_addr.set_port(ph.udp_port as u16);
        self.punch_udp_hole(peer_addr, server, msg_punch, control_permissions).await?;
        return Ok(());
    }

    // ── 3b) TCP 打洞分支 ──
    let mut socket = {
        let socket = connect_tcp(&*self.host, CONNECT_TIMEOUT).await?;   // 连 hbbs（保留 local_addr）
        let local_addr = socket.local_addr();
        // 👇 关键：用同一本地端口主动给 addr_a 发一次 TCP SYN，让 B 的 NAT/防火墙
        //    建立 outbound 映射；之后 A 反向连过来时，NAT 会认为是"已建立连接的回包"，放行。
        //    它必然会失败（A 端可能根本没监听），但这一包 SYN 正是"破洞包"。
        allow_err!(socket_client::connect_tcp_local(peer_addr, Some(local_addr), 30).await);
        socket
    };

    let mut msg_out = Message::new();
    msg_out.set_punch_hole_sent(msg_punch);
    socket.send_raw(msg_out.write_to_bytes()?).await?;     // ← 发 PunchHoleSent 给 hbbs

    crate::accept_connection(server.clone(), socket, peer_addr, true, control_permissions).await;
    Ok(())
}
```

> 🔑 这是整套机制的"魔法"所在：
> 1. B 用某个本地端口主动给 A 的公网地址发一个包（TCP SYN 或 UDP 包），目的不是建立连接，而是
>    **在自己 NAT 表里写入一条到 addr_a 的 outbound 映射**；
> 2. 随后 A 收到 `PunchHoleResponse` 拿到 B 的公网 addr_b，反向发连接请求；
> 3. B 的 NAT 因为已经"见过" addr_a，把这条入站包当成已有连接的回包放行 → 直连成功。

UDP 分支 `punch_udp_hole`（`:693`）做的也是类似的事，只是它会先给 hbbs 发 `PunchHoleSent`，再
**连续 3 次**对 addr_a 发自定义打洞包来确保 NAT 表项建立：

```rust
let (socket, addr) = new_direct_udp_for(&self.host).await?;

        let tm = (hbb_common::time_based_rand() % 20 + 10) as f32 / 1000.;
        hbb_common::sleep(tm).await;
        socket.send_to(&data, addr).await.ok();   // 给 hbbs 发，借此让 NAT 端口保活
    }

```

## 6. 阶段 (5)：hbbs 收 `PunchHoleSent`，转 `PunchHoleResponse` 给 A

`rustdesk-server/src/rendezvous_server.rs:616`：

```rust
async fn handle_hole_sent<'a>(
    &mut self,
    phs: PunchHoleSent,
    addr: SocketAddr,                                // = addr_b（B 端公网映射）
    socket: Option<&'a mut FramedSocket>,            // Some=UDP / None=TCP
) -> ResultType<()> {
    let addr_a = AddrMangle::decode(&phs.socket_addr);
    let mut msg_out = RendezvousMessage::new();
    let mut p = PunchHoleResponse {
        socket_addr: AddrMangle::encode(addr).into(),    // ← B 的公网 addr 给 A
        pk: self.get_pk(&phs.version, phs.id).await,     // ← 服务端用 sk 签名的 IdPk，A 校验对端身份
        relay_server: phs.relay_server.clone(),          // ← 兜底用的 relay
        ..Default::default()
    };
    if let Ok(t) = phs.nat_type.enum_value() {
        p.set_nat_type(t);                                // ← 把 B 的 nat_type 透传给 A
    }
    msg_out.set_punch_hole_response(p);
    if let Some(socket) = socket {
        socket.send(&msg_out, addr_a).await?;             // UDP 直接回 A
    } else {
        self.send_to_tcp(msg_out, addr_a).await;          // TCP 用前面缓存的 sink
    }
    Ok(())
}
```

### 7.1 Relay 回退（`RequestRelay → RelayResponse`）

如果 A 直接判断要走 relay（对端 SYMMETRIC 或 `force_relay`），它走的是另一条路径：先生成 `uuid`，
通过 hbbs 把 `RequestRelay` 转给 B（`rendezvous_server.rs:501`）：

```rust
Some(rendezvous_message::Union::RequestRelay(mut rf)) => {
    if let Some(sink) = sink.take() {
        self.tcp_punch.lock().await.insert(try_into_v4(addr), sink);  // 缓存 A 的 sink
    }
    if let Some(peer) = self.pm.get_in_memory(&rf.id).await {
        let mut msg_out = RendezvousMessage::new();
        rf.socket_addr = AddrMangle::encode(addr).into();          // 把 A 的公网 addr 塞进去
        msg_out.set_request_relay(rf);
        let peer_addr = peer.read().await.socket_addr;
        self.tx.send(Data::Msg(msg_out.into(), peer_addr)).ok();   // UDP 发给 B
    }
}
```

B 收到 `RequestRelay`（`rendezvous_mediator.rs:458`）后调 `create_relay` → 同样连 hbbr，发
`RelayResponse` 回 hbbs，hbbs 在 `rendezvous_server.rs:515` 转给 A。

最终 **A 和 B 都连到 hbbr:21117**，各自带相同 `uuid`：

```rust
// rustdesk-server/src/relay_server.rs:425
async fn make_pair_(stream: impl StreamTrait, addr: SocketAddr, key: &str, limiter: Limiter) {
    if let Ok(Some(Ok(bytes))) = timeout(30_000, stream.recv()).await {
        if let Ok(msg_in) = RendezvousMessage::parse_from_bytes(&bytes) {
            if let Some(rendezvous_message::Union::RequestRelay(rf)) = msg_in.union {
                if !rf.uuid.is_empty() {
                    let mut peer = PEERS.lock().await.remove(&rf.uuid);
                    if let Some(peer) = peer.as_mut() {
                        // 👇 找到了对端：进入双向 relay
                        log::info!("Relayrequest {} from {} got paired", rf.uuid, addr);
                        if !stream.is_ws() && !peer.is_ws() {
                            peer.set_raw();    // 关闭 length-delimited framing，纯字节流
                            stream.set_raw();
                        }
                        relay(addr, &mut stream, peer, limiter, id.clone()).await?;
                    } else {
                        // 还没对上：把自己挂到 PEERS[uuid]，等对端来；30s 超时清除
                        PEERS.lock().await.insert(rf.uuid.clone(), Box::new(stream));
                        sleep(30.).await;
                        PEERS.lock().await.remove(&rf.uuid);
                    }
                }
            }
        }
    }
}
```

`relay` 函数（`:464`）就是个 `tokio::select!` 两端互转，并做带宽/黑名单/降速判定。

### 7.2 本仓库对照点 ✅

- `nat-server/src/relay_server.rs:526` `make_pair_` 实现与上游一致；
- **验证 checklist**：
  - [ ] `PEERS` 必须是 `Mutex<HashMap<String, Box<dyn StreamTrait>>>` 这类内部可变 + 跨任务共享；

## 9. 公钥签名（`get_pk`）—— 防中间人

`rustdesk-server/src/rendezvous_server.rs:1204`：

```rust
async fn get_pk(&mut self, version: &str, id: String) -> Bytes {
    if version.is_empty() || self.inner.sk.is_none() { return Bytes::new(); }
    match self.pm.get(&id).await {
        Some(peer) => {
            let pk = peer.read().await.pk.clone();
            sign::sign(                                                       // ← Ed25519 签名
                &hbb_common::message_proto::IdPk { id, pk, ..Default::default() }
                    .write_to_bytes().unwrap_or_default(),
                self.inner.sk.as_ref().unwrap(),
            ).into()
        }
        _ => Bytes::new(),
    }
}
```

A 端在 `client.rs:577` 用 `secure_connection` 通过 hbbs **公钥** 校验这个签名 → 即使 hbbs 被
冒充也无法伪造一个错误的 pk，因为攻击者没有 hbbs 的 sk。

> 部署侧验证：`id_ed25519` / `id_ed25519.pub` 是 hbbs 首次启动生成的，**不可丢失**——丢失意味着所有
> 客户端要重新接受 pk。本仓库根目录已经有这两个文件，且 `common.rs:gen_sk` 与上游一致。

## 12. 后续可改进点（仅作记录）

1. **`PUNCH_REQS` 去重窗口** —— 本仓库做了 60s 去重，避免 A 重试 3 次导致 B 收到 3 份 `PunchHole`；
   官方 v1.1 也已合入类似机制，不冲突。
2. **`protocol_for_addr` 命中率** —— A、B 编码不同时是真正的"跨编码"，目前实现假设以最近一次 register/punch
   看到的 addr 为准。如果某客户端在短时间内换源端口（如 mobile 4G→Wi-Fi），可能取到旧值，建议加上 TTL。
3. **`RelayResponse` 同样需要 `protocol_for_addr(addr_a)`** —— 检查 `handle_tcp` 的 `RelayResponse` 分支
   是否也走了 codec.serialize。
