RustDesk 在自建服务器（包含 `hbbs` 信令/ID服务器 和 `hbbr` 中转服务器）时，默认需要用到 **21115 到 21119** 这一组端口（Pro版还会用到 21114 和 8000）。

这些端口在整个远程连接、打洞（NAT 穿透）以及中转过程中各司其职。具体端口的作用及分类如下：

## 1. HBBS（ID/信令服务器）相关端口

`hbbs` 是基础调度中心，负责客户端的注册、在线状态心跳、以及帮助两个客户端互相发现。

- **21115 / TCP —— NAT 类型测试**
  - **作用：** 专门用于检测客户端当前的网关（NAT）类型。客户端在启动时，会通过这个端口向服务器发送探测包，以此判断自己处于什么网络环境下（如对称型 NAT 还是锥形 NAT），从而决定后面采用哪种穿透策略。
- **21116 / UDP —— ID 注册与心跳服务（最关键）**
  - **作用：** 客户端保持在线的通道。客户端启动后，会通过这个 UDP 端口向服务器注册自己的 ID，并定期发送“心跳包”维持连接。**这也是内网穿透（P2P 打洞）的核心端口**，两个客户端想要直接连通，必须通过该 UDP 端口进行打洞协商。
- **21116 / TCP —— TCP 打洞与连接协调**
  - **作用：** 当进行 TCP 穿透或者需要通过信令建立初次握手时，客户端会连接服务器的这个 TCP 端口。

## 2. HBBR（中转服务器）相关端口

当两个客户端网络环境太复杂（比如都是校园网、公司内网或对称型 NAT），**打洞失败无法直连时，所有流量都会走 `hbbr` 服务器中转**。

- **21117 / TCP —— 远程桌面中转连接**
  - **作用：** 真正的中转通道。当 P2P 直连失败，被控端和主控端会同时连接服务器的 21117 端口，由服务器在中间转发屏幕画面、鼠标键盘指令以及文件传输流。
- **21119 / TCP —— 网页版客户端中转（WebSocket）**
  - **作用：** 专门给 **RustDesk Web Client（网页版客户端）** 提供的中转支持。它走的是 WebSocket 协议，允许用户直接在浏览器里控制远程电脑。

## 3. Web 端与管理服务端口（普通自建可不开启）

如果你使用的是开源版，且不用网页版客户端，以下端口可以不暴露：

- **21118 / TCP —— 网页版客户端信令**
  - **作用：** 对应 `hbbs` 的网页版支持。当使用浏览器登录 RustDesk 网页端时，通过这个端口进行 ID 注册和信令交互。
- **21114 / TCP —— Web 管理后台（仅限 Pro 专业版）**
  - **作用：** 专业版附带的 Web 控制台端口，用于设备管理、用户权限审计等。
- **8000 / TCP —— 自动配置/安装包下载端口（可选）**
  - **作用：** 部分一键安装脚本会带有一个简单的 HTTP 服务器，用来通过网页直接下载已经打包好服务器 IP 和 Key 的客户端安装包。

## 总结汇总表

| **端口号 / 协议** | **所属服务** | **核心作用**                        | **自建基本款是否必开？**   |
| ----------------- | ------------ | ----------------------------------- | -------------------------- |
| **21115 / TCP**   | `hbbs`       | NAT 环境类型检测                    | **是**                     |
| **21116 / UDP**   | `hbbs`       | ID 注册、保持在线心跳、**P2P 打洞** | **是（核心，必开）**       |
| **21116 / TCP**   | `hbbs`       | 信令协调、TCP 打洞支持              | **是**                     |
| **21117 / TCP**   | `hbbr`       | 画面与文件传输的**中转通道**        | **是（直连失败时的保障）** |
| **21118 / TCP**   | `hbbs`       | 网页端客户端（Web Client）信令      | 否（不用网页版可不开）     |
| **21119 / TCP**   | `hbbr`       | 网页端客户端（Web Client）中转      | 否（不用网页版可不开）     |

> 💡 **排错小提示：**
>
> 很多人在云服务器（如腾讯云、阿里云）安全组或防火墙里开放端口时，**经常漏掉 21116 的 UDP 协议**（只开了 TCP）。这会导致客户端虽然显示“就绪（Ready）”，但是两台电脑连接时永远卡在“正在连接”，或者死活连不上，因为它们没办法通过 UDP 进行打洞和身份握手。



21117端口可以做其它第三方应用的数据轩发吗？



**不可以，直接把 21117 端口当成通用的 TCP 端口转发工具（比如 Nginx、socat 那样）是行不通的。**

因为这个端口并不是一个纯粹的“盲转”端口，它后面运行着 RustDesk 的 `hbbr`（中转服务）程序。

## 为什么不能直接转发第三方数据？

`hbbr` 服务在 21117 端口上运行了一套**私有的应用层协议**（类似于一种暗号或握手协议）：

1. **身份校验：** 当客户端连接 21117 端口时，它必须先发送符合 RustDesk 规范的特定握手报文，里面包含连接令牌（Token）、目标 ID、加密密钥（Key）等信息。
2. **配对机制：** `hbbr` 收到主控端的请求后，必须在内存里找到对应的被控端连接，然后将这两路连接“桥接”在一起。

如果你尝试把第三方应用（比如网页 HTTP 流量、SSH 流量、或者数据库连接）直接打到 21117 端口，`hbbr` 根本听不懂这些数据在说什么，**会直接判定为非法数据包并断开连接**。

## 如果你真的想借用 RustDesk 的网络能力转发第三方流量，该怎么办？

虽然你不能直接往 21117 端口灌数据，但如果你想利用 RustDesk 已经搭建好的内网穿透网络，来访问内网的其它服务（比如内网的 SSH、数据库、RDP 桌面），RustDesk 官方其实提供了一个隐藏功能：**端口转发（Port Forwarding）**。

### 官方的“借道”实现方案

RustDesk 客户端内置了类似 SSH Tunneling（端口隧道）的功能。你可以通过 RustDesk 客户端进行配置，让它帮你把第三方流量打包，伪装成远程桌面的流量走 21117 端口中转：

1. **在主控端客户端：** 找到对应的远程电脑卡片，点击菜单里的 **“端口转发” (Port Forwarding)**。
2. **配置映射规则：**
   - 设置本地端口（Local Port）：例如 `1234`
   - 设置目标 IP（Remote Host）：例如 `127.0.0.1`（代表被控端本机）或内网其它 IP
   - 设置目标端口（Remote Port）：例如 `3389`（Windows 自带远程桌面）或 `22` (SSH)
3. **使用第三方应用连接：** 配置完成后，你只需要在主控端电脑上连接 `127.0.0.1:1234`，流量就会被 RustDesk 客户端安全地压缩、加密，通过 `hbbr` 的 21117 端口安全地运送到被控端内网。

## 总结

- **直接把 21117 当通用转发器：** ❌ 绝不可能，协议不匹配，数据会被丢弃。
- **利用 RustDesk 客户端的“端口转发”功能间接转发：**  通过这种方式，21117 端口实际上成了你所有第三方内网流量的安全隧道。

## 核心操作步骤

在操作前，请确保主控端和被控端都已经正常登录并在 RustDesk 列表中。



1

找到目标设备并打开菜单

主控端操作

在主控端电脑上打开 RustDesk，在列表中找到你要连接的那台远程电脑（被控端）。点击卡片右上角的 **三个点（菜单）** 按钮，在弹出的选项中选择 **“端口转发” (Port Forwarding)**。

2

添加新的转发规则

规则配置

在弹出的端口转发窗口中，点击 **“添加” (Add)** 按钮，会看到三个核心输入框：

- **本地端口 (Local Port)：** 在你**主控端**电脑上随便找个没被占用的空闲端口，比如 `1234` 或 `2222`。
- **远程主机 (Remote Host)：** 填写相对于**被控端**而言的 IP。如果就是要连被控端电脑本身，填 `127.0.0.1`；如果要连被控端所在内网的另一台设备（如群晖 NAS），就填它的内网 IP（如 `192.168.1.5`）。
- **远程端口 (Remote Port)：** 你想要访问的目标服务端口。连 SSH 就填 `22`；连 Windows 远程桌面就填 `3389`。

3

保存并开启规则

启动隧道

配置完成后点击保存。回到转发规则列表，勾选刚刚创建的规则，或者点击启用按钮。此时主控端会在本地监听你设置的那个本地端口（比如 `1234`）。

4

使用第三方工具建立连接

连接测试

现在你可以打开你的第三方工具了。比如你想连 SSH，不用管远程公网 IP 是什么，直接在终端里连接本地： `ssh username@127.0.0.1 -p 1234` RustDesk 会自动拦截这个 `1234` 端口的流量，通过 21117 端口的安全隧道发送到被控端。



## 两个经典应用场景配置示例

### 场景 A：借道连接 Windows 自带的 RDP 远程桌面 (3389)

虽然 RustDesk 本身就能看画面，但有时候 Windows 原生的 RDP 体验在画质和丝滑度上更好。

| 配置项          | 填写内容    | 说明                     |
| --------------- | ----------- | ------------------------ |
| **Local Port**  | `33890`     | 主控端本地随便起一个端口 |
| **Remote Host** | `127.0.0.1` | 代表被控端电脑自己       |
| **Remote Port** | `3389`      | Windows RDP 默认端口     |

- **如何连接：** 在主控端按下 `Win + R` 输入 `mstsc` 打开远程桌面连接，计算机栏输入 `127.0.0.1:33890` 即可直连。

### 场景 B：访问被控端内网里的群晖 NAS 管理后台 (5000)

假设被控端是你父母家的一台 PC，它和群晖 NAS 在同一个路由器下。

| 配置项          | 填写内容        | 说明                           |
| --------------- | --------------- | ------------------------------ |
| **Local Port**  | `5000`          | 主控端本地端口                 |
| **Remote Host** | `192.168.1.100` | **群晖 NAS 在其内网的真实 IP** |
| **Remote Port** | `5000`          | 群晖默认的 HTTP 端口           |

- **如何连接：** 在主控端浏览器里直接输入 `[http://127.0.0.1:5000](http://127.0.0.1:5000)`，就能直接看到父母家内网里那台群晖的登录界面。

> 💡 **关键细节提示：**
>
> - **无需维持远程画面：** 端口转发功能启用后，即使你关闭了 RustDesk 的远程桌面大窗口，只要 RustDesk 主程序还在后台运行，这个数据隧道就一直有效。
> - **安全性极高：** 所有通过这个通道传输的第三方流量，都会自动继承 RustDesk 的底层非对称加密（Key 机制），在传输过程中哪怕走的是公网中转（21117 端口），别人也无法窃听。
